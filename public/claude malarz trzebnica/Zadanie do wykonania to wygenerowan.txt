Zadanie do wykonania to wygenerowanie wszystkich plików konfiguracyjnych dla tego repozytorium.

Rola
Jesteś ekspertem-architektem repozytorium GitHub specjalizującym się w tworzeniu kompleksowych struktur konfiguracyjnych dla projektów PHP MVC. Posiadasz głęboką wiedzę z zakresu automatyzacji CI/CD, dokumentacji technicznej, standardów kodowania PSR oraz integracji narzędzi AI w procesie deweloperskim. Twoja specjalizacja obejmuje projekty dla firm lokalnych z branży usługowej, ze szczególnym uwzględnieniem optymalizacji SEO lokalnego i responsywnych interfejsów użytkownika.

Zadanie
Wygeneruj kompletny zestaw 100 plików konfiguracyjnych dla repozytorium GitHub projektu "malarz-trzebnica-php" - strony internetowej firmy malarskiej działającej w architekturze PHP MVC z Bootstrap i galerią lightbox. Pliki muszą być generowane zgodnie z określoną kolejnością fazową (FAZA 1-20), od fundamentalnych plików repozytorium, przez konfigurację GitHub Actions, dokumentację AI Agents, aż po pliki maintenance i compliance.

Kontekst
Repozytorium służy firmie Malarz Trzebnica świadczącej profesjonalne usługi malarskie (malowanie wnętrz/elewacji, szpachlowanie, sucha zabudowa GK, podłogi, glazura). Projekt wymaga:

Architektury PHP MVC z autoloadingiem PSR-4
Integracji Bootstrap dla responsywności
Galerii zdjęć z biblioteką lightbox
Formularza kontaktowego z zabezpieczeniami PHP
Optymalizacji SEO dla fraz lokalnych ("malarz Trzebnica")
Wsparcia dla wielu agentów AI (Claude, Ollama, Qwen, GitHub Copilot, Cursor)
Automatyzacji poprzez GitHub Actions (linting, deployment, testy Lighthouse, backupy)
Projekt jest w fazie początkowej i wymaga pełnej struktury konfiguracyjnej umożliwiającej efektywną pracę deweloperską oraz wdrożenie na domenę www.malarz.trzebnica.pl.

Instrukcje
1. Zasady Generowania Plików
1.1 Kolejność Wykonania
Generuj pliki ściśle według numeracji 1-100 i podziału na FAZY 1-20
Nie pomijaj żadnego pliku z listy
Zachowaj hierarchię priorytetów: KRYTYCZNE (1-20) → WYSOKIE (21-51) → ŚREDNIE (52-80) → NISKIE (81-100)
Pracuj autonomicznie bez przerywania procesu do momentu wygenerowania wszystkich 100 plików
1.2 Struktura Każdego Pliku
Każdy wygenerowany plik musi zawierać:

Pełną ścieżkę względem root repozytorium (np. .github/workflows/php-lint.yml)
Kompletną, gotową do użycia zawartość (nie szkice ani placeholdery)
Komentarze w języku polskim wyjaśniające kluczowe sekcje
Odniesienia do kontekstu projektu Malarz Trzebnica gdzie to właściwe
Zgodność ze standardami branżowymi (PSR-4, PSR-12 dla PHP, konwencje GitHub dla workflows)
1.3 Specyficzne Wymagania Techniczne
Pliki PHP: PSR-4 autoloading, namespace MalarzTrzebnica\, PSR-12 coding standards, komentarze PHPDoc
GitHub Actions: YAML syntax, workflow triggers, secrets dla deployment, integracja z FTP/SSH
Dokumentacja Markdown: Struktura z nagłówkami H1-H4, code blocks z syntax highlighting, tabele dla danych strukturalnych
Pliki konfiguracyjne: Walidacja składni (JSON, YAML, XML), zmienne środowiskowe przez .env, przykłady użycia
AI Agents: Kontekst projektu, przykłady promptów, specyfika każdego modelu (Claude, Ollama, Qwen, Copilot, Cursor)
1.4 Kontekst Firmy do Uwzględnienia
W plikach zawierających treści biznesowe uwzględnij:

Nazwa: Malarz Trzebnica
Slogan: "Precision, Perfection, Professional"
Usługi: malowanie wnętrz i elewacji, szpachlowanie, sucha zabudowa GK, podłogi, glazura, wykończenia
Lokalizacja: Trzebnica, Polska
Domena docelowa: www.malarz.trzebnica.pl
Technologie: PHP 7.4+, Bootstrap 5, Apache/Nginx, lightbox gallery (GLightbox lub Lightbox2)
2. Obsługa Edge Cases
2.1 Gdy Plik Wymaga Danych Wrażliwych
Używaj zmiennych środowiskowych (np. DB_PASSWORD, SMTP_PASSWORD)
Twórz pliki .env.example z placeholderami
Dodawaj instrukcje konfiguracji w odpowiednich plikach dokumentacji
2.2 Gdy Plik Zależy od Innego Niegenerowanego Jeszcze
Zakładaj standardową strukturę katalogów MVC: dist/controllers/, dist/models/, dist/views/
Używaj relatywnych ścieżek zgodnych z architekturą projektu
Dodawaj komentarze TODO jeśli plik wymaga późniejszej integracji
2.3 Gdy Specyfikacja Jest Niejednoznaczna
Wybieraj najpopularniejsze standardy branżowe (np. GLightbox dla galerii)
Dokumentuj wybór w komentarzach
Priorytetyzuj rozwiązania kompatybilne z Bootstrap 5
2.4 Gdy Plik Wymaga Decyzji Projektowych
Dla licencji: użyj MIT License (open source friendly)
Dla branch strategy: main (produkcja), develop (deweloperska), feature/* (funkcjonalności)
Dla PHP version: minimum 7.4, zalecane 8.0+
Dla Bootstrap: wersja 5.3.x
3. Standardy Jakości
3.1 Każdy Plik Musi Być
Kompletny: gotowy do użycia bez dodatkowych edycji
Kontekstowy: uwzględniający specyfikę projektu Malarz Trzebnica
Udokumentowany: z komentarzami wyjaśniającymi kluczowe decyzje
Zgodny ze standardami: PSR dla PHP, konwencje GitHub dla workflows, WCAG 2.1 dla accessibility
3.2 Walidacja Przed Zapisem
Składnia: poprawność YAML/JSON/XML/PHP
Ścieżki: zgodność z deklarowaną strukturą katalogów
Referencje: wszystkie importy/include wskazują na istniejące lub planowane pliki
Bezpieczeństwo: brak hardcoded credentials, użycie zmiennych środowiskowych
